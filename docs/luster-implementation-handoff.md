## P7.4 — Smart Fit owner settings + permission enforcement — COMPLETE (2026-07-19)

> **Owners can now enable, configure, and disable Smart Fit themselves.** Settings → Booking → **Smart Fit discounts** (SettingsModal view `'smart-fit'`, new self-contained `src/components/admin/SmartFitSettingsCard.tsx`). No reporting dashboard, no revenue attribution, no evaluator/pricing/checkout changes, no migration, no deployment — **explicitly deferred to P7.5:** Smart Fit reporting, revenue attribution, aggregate usage metrics, final end-to-end production QA, deployment readiness, and deployment. The identity-aware availability seam from P7.3 also remains open.

**Persisted shape & semantics** (unchanged contract, `salon.settings.smartFit`, parsed by `resolveSmartFitConfig`/`readStoredSmartFitSettings`): `{enabled, discountType: 'percent'|'fixed', value (whole % | cents), maxRemainingGapMinutes, minImprovementMinutes, eligibleServiceIds[], eligibleTechnicianIds[]}`. **Empty `eligibleServiceIds`/`eligibleTechnicianIds` = ALL services/technicians eligible** — the UI copy states this exactly ("Leave every service unchecked to allow Smart Fit on all services."). Defaults when nothing is stored (display only, nothing persisted by merely visiting): off, percent 10, gap 10, improvement 20, empty lists. Missing/invalid config still renders (and stays) safely disabled — visiting or saving unrelated settings can never activate the feature.

**Server** (`/api/admin/salon/settings` GET+PATCH): new strict write-side `smartFitSettingsUpdateSchema` + `mergeSmartFitSettings` in `src/libs/smartFitConfig.ts` — integers only inside `SMART_FIT_LIMITS` (percent 1–100, fixed 1–5,000,000¢, gap 0–60, improvement 0–240; no NaN/negative/fractional/stringly values can persist), field-by-field merge so `{enabled:false}` preserves the rest of the config and re-enabling restores it verbatim, id arrays deduped, and a cross-field guard (switching a stored fixed-cents value to percent 400s instead of persisting a >100% discount). Ids supplied in an update are ownership-validated against `service`/`technician` rows of THIS salon (400 `INVALID_SERVICE`/`INVALID_TECHNICIAN`; ownership only, not isActive, so an archived-but-owned stale id never bricks a save; stored ids from earlier saves are not re-validated, so a deleted service can't block unrelated saves). Smart-fit-only saves write via single-key `jsonb_set` (payments/merchandising precedent — concurrent saves of other keys are never clobbered); responses read the saved row back through the shared parser. **Permissions:** the existing `requireAdmin` guard (salon-membership-scoped, server-resolved salon from slug, never client-trusted salon ids) gates both GET and PATCH — same bar as every other salon setting; staff/unauthenticated get 401/403 on view and save; cross-salon ids are rejected.

**Audit:** the existing `settings_updated` audit event (via `logAuditEvent`, same as all settings saves) records actor, salon, timestamp, and `metadata.{before,after}.smartFit` — enabled-state transitions and changed values are reconstructable; no customer data, availability payloads, or secrets are stored. No new audit subsystem was built (none needed; per-grant `appointment_audit_log` rows already land at booking time from P7.2).

**Owner UI** (`SmartFitSettingsCard`, explicit-save payments pattern): plain-language intro ("Smart Fit can offer customers a discount for choosing times that reduce gaps in your schedule."), master toggle (default off) with a visible inactive note when off (settings kept but inactive; existing bookings keep their discount), percentage/fixed selector with %- or currency-suffixed input (salon currency via `bookingConfig.currency`, fixed dollars→cents on save), display-only example line via the shared `calculateSmartFitDiscountCents` ("Example: A $65.00 service … would be $58.50."), minute inputs with the approved plain-language descriptions, service + technician checkbox pickers (active rows from `/api/salon/services` + `/api/admin/technicians?status=active`, Select all / Clear selection, long names wrap, 44px touch targets), stale-saved-id surfacing (counted note; stale ids ride along in the payload untouched — pruning could silently flip an emptied list to "all eligible") and a "How Smart Fit works" plain-language help block (server-confirmed offers only; campaign/reward/first-visit priority; never moves appointments; customers confirm their time; disabling stops new offers without touching existing bookings). Save: loading/disabled state, duplicate-submission guard, truthful success only after 200, failed saves keep every entered value with a retry-able error, client validation with inline `role=alert` errors + `aria-invalid`/`aria-describedby` + focus-to-first-invalid; dirty state feeds the modal's unsaved-changes Back warning.

**Feature-dark verified:** no valid enabled config ⇒ no annotation (unchanged parser default); saving a disabled config keeps it dark; enabling via the UI activates the existing P7.2/P7.3 flow (no changes to evaluator/overlay/booking); disabling stops new offers; booked discount snapshots are never recalculated (P7.2 locked-snapshot columns untouched).

**Public slot grouping (time step).** When the loaded availability has server-marked eligible slots: a "Save with a Smart Fit" section renders FIRST (approved heading + "Choose a time that fits neatly into the salon's schedule and save on your appointment."), then an "Other available times" heading over the existing morning/afternoon grids. Chronological order preserved in both groups; a qualifying slot never renders twice (it is removed from the regular grids); booked/blocked slots keep their legacy disabled presentation in the regular grids. Zero qualifying slots ⇒ byte-equivalent legacy layout (no headings); Smart Fit disabled ⇒ no annotation ⇒ legacy. Campaign links (`?campaign=`) null the annotations at parse time (winner-take-all — never promise a saving beside a campaign offer). Date/technician/service changes regroup via the normal availability refetch.

**Slot presentation.** Each qualifying slot card: 12h time, "Smart Fit" badge (text badge — not color-alone; `text-neutral-900` on a primary-tinted pill for WCAG AA), "Save $X.XX" line, discounted price as the most prominent price, struck-through original (`text-neutral-600`, AA-checked); all money through `formatMoney`. Full accessible name on the button (`10:30 AM — Smart Fit: save $6.50. $58.50 instead of $65.00.`) with the visual internals `aria-hidden`. No urgency copy, no config details, no evaluator language.

**Nearby suggestion (confirm step).** Selecting a regular slot computes at most ONE suggestion — derived exclusively from the already-loaded availability response (same date/technician-context/location by construction; no extra request): server-eligible, not booked, not the selected slot, smallest absolute time difference, exact tie → earlier. It travels to the confirm page as URL params; the banner reads "Save $6.50 by booking 30 minutes earlier" + "1:30 PM · $58.50 instead of $65.00" with **Choose this time** (router.replace to the suggested slot with its expectation params — pricing updates from the same availability data, banner closes, all other choices preserved) / **Keep my time** (selection untouched). **Dismissal.** "Keep my time" stores the dismissal in sessionStorage (the established per-tab booking-state pattern; no DB field, no localStorage) keyed by the full booking context (salon, date, technician, location, base service/services, add-ons — order-insensitive). Same context ⇒ stays dismissed across re-selection and remounts; ANY material change (service, add-ons, technician, location, date) resets it (the stored record is cleared on first mismatch, so returning to the original context re-offers). Legacy confirm mounts (no suggestion params) never touch the dismissal store. The "Save with a Smart Fit" section itself never hides on dismissal.

**Selection/review/submission.** A selected Smart Fit slot carries only `smartFitDiscountCents`/`smartFitTotalCents` to the confirm page; the review activates the offer ONLY when no higher-priority discount preview applies (campaign/first-visit/`discountAmount>0`, plus the session "outranked" flag below) AND the params reconcile exactly with the RSC-computed subtotal (tampered/stale URLs degrade to regular pricing — a URL can never lower a price). Active offer: BookingCard total, confirm button, and success screen show the formatted discounted total; a "Smart Fit Discount applied" review card shows `Subtotal $65.00 · Smart Fit Discount −$6.50 · Total $58.50` (display-only; nothing editable). The POST attaches exactly the two P7.2 fields `expectedDiscountType:'smart_fit'` + `expectedTotalCents`; regular slots submit neither; tax/deposits stay booking-absent by design (tax enters at checkout via the one P7.0/P3 pipeline). Reschedule flows reuse the same params (self-exclusion already server-side).

**Stale handling (409 SMART_FIT_CHANGED).** No booking is created or presented. A full-screen SlotTakenState-pattern alert (role=alert, focus moved to the action) shows the server's exact message ("This discounted time is no longer available. Please choose from the latest times." — fallback constant byte-identical) and "Choose another time" → sets a salon-scoped one-shot refresh flag → router.back() to the time step, which refetches availability and moves focus to the refreshed time list; services/add-ons/technician stay in the URL, contact details stay in sessionStorage (verified live), the stale time is cleared (no selection state persists), nothing silently reselects or resubmits. **Precedence self-heal:** when the 409 breakdown proves a higher-priority discount replaced Smart Fit (`discountType` ≠ smart_fit with a positive amount — e.g. a first-visit-eligible guest whose identity availability cannot see), the alert additionally displays the returned final pricing honestly ("Current price for this time: $48.75 (First visit discount)") and a salon-scoped session flag suppresses all further Smart Fit promises/expectations for that visitor — closing the adversarially-found guest 409 loop (identity-blind availability would otherwise re-promise the same un-honorable saving forever).

**Known limitation (deferred to P7.4+, needs a server seam):** availability annotation is identity-blind, so a logged-in client whose reward/first-visit outranks Smart Fit can still see time-step savings that the confirm step then (correctly, honestly, without a 409) replaces with their better/other pricing; full fix = identity-aware availability annotation. The confirm-step gate guarantees no un-honorable expectation is ever submitted in that path.

**Accessibility & responsive.** Verified: badge/savings text (not color/strikethrough alone), full accessible names on slot and suggestion buttons, keyboard-operable suggestion + dismissal, role=alert stale pattern with focus to the action, focus-to-refreshed-times after stale return, labeled regions for all three slot sections, AA contrast on the new elements, sr-only outcome announcements. ## P7.1 — Smart Fit config + pure eligibility evaluator — COMPLETE (2026-07-19)

> **Smart Fit remains NOT customer-facing and NOT fully implemented.** P7.1 ships only the two foundation libraries from the P6 plan — nothing imports them yet: no availability/booking-write integration, no `applySmartFitOverlay` (the P6 phase list assigns only "pure evaluator + config" to P7.1; the overlay lands with the booking-POST authority phase), no UI, no settings surface, no reporting, no audit events, **no migration** (settings ride `salon.settings.smartFit` JSONB; `discountType` stays free text).

**New files:** `src/libs/smartFitConfig.ts` (+test, 8), `src/libs/smartFit.ts` (+test, 36). One type-only addition: `smartFit` namespace on `SalonSettings` (`src/types/salonPolicy.ts`). No app file imports either module yet — behavior is provably unchanged (full suite green with zero existing-test edits).

**Configuration (`resolveSmartFitConfig`, mirrors taxConfig.ts).** Stored shape (approved settings ONLY — the dropped before/after/between toggles, stacking, and suggestion-distance settings do not exist and unknown fields are stripped): `{enabled, discountType: 'percent'|'fixed', value, maxRemainingGapMinutes, minImprovementMinutes, eligibleServiceIds[], eligibleTechnicianIds[]}`. Defaults: **OFF**, percent 10, gap 10, improvement 20, empty lists = all services/techs. Resolution never throws: missing/malformed namespaces (wrong-typed fields fail the whole safeParse) collapse to the inert `DISABLED_SMART_FIT_CONFIG`; numerics truncate to integers and clamp to `SMART_FIT_LIMITS` (percent ≤ 100, fixed ≤ 5,000,000¢, gap ≤ 60, improvement ≤ 240); empty ids are filtered. `readStoredSmartFitSettings` returns the raw shape for the future editing surface. `calculateSmartFitDiscountCents` mirrors `calculateRetentionDiscount`: percent floors, both modes clamp to [0, subtotal].

**Evaluator (`evaluateSmartFitSlot` — pure, deterministic, zero DB/`Date.now()`/timezone math).** Inputs are caller-resolved UTC-ms values on the slot-grid basis (`zonedTimeToUtc` + `bookingConfig.timezone` — NOT bookingPolicy's Toronto minutes math): a `SmartFitDayContext` (technician, location, effective working window, day blocks, slot interval, grid anchor = salon-local midnight, `nowMs`, lead-time default 120) and a `SmartFitCandidate` (start, visible service+add-on minutes, buffer, service/tech/location, opaque `clientKeys`, reschedule `excludeAppointmentId`). Blocks carry `{id, kind: appointment|break|time_off|google_busy, startMs, endMs (BLOCKED end incl. that appointment's buffer), clientKeys?}`. Gate order (first failure wins): enabled → service eligibility → technician eligibility → explicit technician mismatch (`null` = any, matches the day context) → explicit location mismatch → minimum notice → working window → overlap (blocked window vs blocked windows, both buffers included).

**Eligibility formula (as approved in P6):** free span `L = (prevEdge, nextEdge)` where `prevEdge = max(workStart, latest block end ≤ candidate start)` and `nextEdge = min(workEnd, earliest block start ≥ candidate blocked end)` — **all** kinds shrink the span (Google included). **Slack rule (anti-trivial-switch):** `max(prevEdge, now + leadTime) → nextEdge` minus the blocked duration must be ≥ `minImprovementMinutes`; exact fill (slack 0) never qualifies (no placement freedom to reward), and unbookable pre-lead-time room never counts. **Exists-side rule:** ≥ 1 side must have (a) an edge formed by a QUALIFYING block — appointment/break/time_off; never a working-window boundary, never Google (salon-wide freeBusy can't prove per-tech adjacency) — (b) gap ≤ the grid-clamped max, (c) the candidate as the tightest FEASIBLE grid slot on that side (`before: start == ceilToGrid(max(edge, leadBoundary))`; `after: start == floorToGrid(nextEdge − blockedDuration)` — off-grid candidates fail here), and (d) **self-adjacency exclusion**: at least one non-self qualifying block at the edge (`clientKeys` intersection; a client's own appointment cannot mint their discount — book-filler-then-cancel closed; `excludeAppointmentId` separately removes the client's CURRENT appointment entirely for reschedules). **Grid-quantization clamp (exact formula, in code + tests):** `effectiveMaxGap = max(configuredMaxGap, slotIntervalMinutes − 5)` — blocked ends land off-grid, so a 30-min-grid salon's tightest reachable gap can be 25 minutes through rounding alone; the tightest-feasible rule stops the clamp from rewarding loose fits.

**Result (structured, privacy-clean):** `{eligible, reason, sides.{before,after}: {edge: block|boundary|none, neighbor: {id, kind}|null, gapMinutes, tightest, selfOnly, qualifies}, qualifyingSides, remainingGapMinutes, improvementMinutes (slack), consolidatedMinutes (slack − remainingGap: the far side stays one contiguous span), effectiveMaxGapMinutes}`. Neighbor references are internal id+kind only — `clientKeys` and any client details never appear in results (asserted by test). **Stable reason codes** (frozen by test): `ELIGIBLE, SMART_FIT_DISABLED, SERVICE_NOT_ELIGIBLE, TECHNICIAN_NOT_ELIGIBLE, TECHNICIAN_MISMATCH, LOCATION_MISMATCH, MINIMUM_NOTICE, OUTSIDE_WORKING_HOURS, OVERLAPS_BLOCK, INSUFFICIENT_IMPROVEMENT, SELF_ADJACENCY, NOT_TIGHTEST_SLOT, GAP_TOO_LARGE, NO_QUALIFYING_NEIGHBOR` (ineligible-side reporting picks the most informative: self > not-tightest > gap > no-neighbor).
